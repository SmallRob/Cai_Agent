**问题提出者**：Weizhenlong

## 1. 问题背景

### 1.1 当前挑战

在行内大型代码库（200 万行）的项目中，现有的上下文构建方案存在以下问题：

| 问题 | 表现 |
|------|------|
| **全量生成不完整** | 直接在根目录运行上下文构建时，AI 生成的内容不全面 |
| **拆分过细效率低** | 如果拆模块拆得太细，构建上下文时间过长 |
| **上下文使用不理想** | AI 使用上下文执行后续任务时，因为内容过多导致效果不理想 |

### 1.2 现有工具集

项目中已有的上下文构建工具：

- **Context Builder**（基础版）：一次性生成整个项目的上下文
- **Project Context Analysis**（增强版）：按模块逐个生成上下文，支持状态追踪

### 1.3 核心问题

> **200 万行代码库，上下文究竟应该拆到多细才是合理的？**

## 2. 参考框架

### 2.1 ASDM Toolset 设计原理

Toolset 是 AI Coding 工具的能力扩展框架，采用以下架构：

```
.asdm/toolsets/<toolset-id>/
├── README.md                    # 工具集说明
├── INSTALL.md                   # 安装指南
├── actions/                     # 动作指令目录
│   └── <action-name>.md         # 各动作的具体指令
└── spec/                        # 规范模板目录
    └── <spec-name>-spec.md      # 各类型文档的模板
```

**核心组件职责**：

| 组件 | 作用 |
|------|------|
| README.md | 工具集概述、功能说明、使用方法 |
| actions/ | 定义 AI 执行的具体动作流程 |
| spec/ | 定义输出文档的结构模板 |

### 2.2 上下文注入机制

ASDM 采用**渐进式上下文加载策略**，避免一次性加载过多上下文：

| 阶段 | 加载内容 | 说明 |
|------|----------|------|
| 初始阶段 | index.md | 必读，了解项目整体结构 |
| 规划阶段 | 按需加载 | 根据功能需求加载相关上下文 |
| 执行阶段 | 按需加载 | 根据任务需求加载相关上下文 |

**上下文使用原则**：

1. **必读 index.md**：始终首先读取项目入口文件
2. **按需加载**：只加载当前任务需要的上下文
3. **避免过载**：不一次性加载所有上下文文件
4. **相关性**：只请求与当前任务直接相关的上下文

### 2.3 PRD Builder 的上下文注入示例

```markdown
## 上下文注入

### 需要读取的上下文文件

1. **index.md**（必需 - 必须首先读取）
   - 路径：`.asdm/contexts/index.md`
   - 目的：提供工作空间的概览

2. **渐进式上下文阅读**（可选 - 按需）
   - 仅在查看 index.md 后，如果需要额外上下文时
   - AI 模型可以根据正在规划的功能请求特定的上下文文件

### 渐进式上下文阅读策略

1. **初始阶段**（规划开始前）：
   - 仅读取 `.asdm/contexts/index.md`
   - 用它来了解整体工作空间结构并识别相关领域

2. **规划阶段**（任务规划期间）：
   - 确定哪些额外上下文文件与该功能相关
   - 根据需要一次请求一个特定的上下文文件

3. **执行阶段**（任务执行期间）：
   - 在实现过程中可以根据需要引用或重新读取上下文文件
```

## 3. AI 上下文工作机制分析

### 3.1 注意力衰减与"中间丢失"问题

```
[上下文窗口示意]
┌─────────────────────────────────────────────────────────┐
│  开头内容    ████████░░░░░░░░░░░░░░░░░░░░░░░░░░░  结尾内容  │
│  (高权重)          ↓注意力递减↓                (低权重)   │
└─────────────────────────────────────────────────────────┘
```

- AI 对输入的注意力是**非均匀分配**的
- 距离当前任务位置越远的内容，注意力权重越低
- **超长上下文会导致"中间丢失"**：中间部分的信息被两头稀释

### 3.2 信息密度 vs 绝对信息量

| 指标 | 重要性 |
|------|--------|
| 信息密度 | ★★★★★ |
| 与任务的相关性 | ★★★★★ |
| 绝对信息量 | ★★☆☆☆ |

> **关键洞见**：上下文中的冗余信息不仅无益，还会**稀释**关键信息的权重。

### 3.3 渐进式上下文加载模式

```
┌─────────────────────────────────────────────────────────────┐
│                    渐进式上下文加载                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Phase 1: 入口加载                                          │
│  ┌─────────┐                                                │
│  │ index   │  → AI 建立项目整体认知，找到导航                │
│  └─────────┘                                                │
│       ↓                                                     │
│  Phase 2: 任务相关上下文加载                                 │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐                   │
│  │模块索引  │ + │数据模型  │ + │编码规范 │  → 按需组合       │
│  └─────────┘   └─────────┘   └─────────┘                   │
│       ↓                                                     │
│  Phase 3: 执行时按需补充                                     │
│  ┌─────────┐   ┌─────────┐                                 │
│  │API详情   │ + │源码引用  │  → 遇到具体实现时加载           │
│  └─────────┘   └─────────┘                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. 合理上下文粒度设计原则

### 原则一：按"决策点"粒度，而非按代码结构粒度

**错误思路**：按模块/子模块/文件层级拆分
**正确思路**：按 AI 做决策时需要的知识粒度拆分

```
❌ 不合理的粒度
├── module-A/
│   ├── submodule-A1/      # 太细，AI无法形成整体认知
│   │   ├── component-x
│   │   └── component-y
│   └── submodule-A2/

✓ 合理的粒度（按"决策域"）
├── 领域/
│   ├── 用户域/            # 同一个业务领域的上下文放一起
│   ├── 订单域/
│   └── 支付域/
├── 基础设施/              # 技术基础设施独立
│   ├── 数据库访问/
│   ├── 缓存/
│   └── 消息队列/
└── 跨领域/                # 横切关注点
    ├── 认证授权/
    └── 日志监控/
```

### 原则二：上下文粒度应与任务粒度匹配

```
┌─────────────────────────────────────────────────────────────┐
│              任务粒度 ←→ 上下文粒度 匹配矩阵                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  任务类型              需要的上下文粒度                      │
│  ─────────────────────────────────────────────              │
│  1. 理解项目全貌       项目索引（< 100 行）                  │
│  2. 规划新功能         相关领域索引 + 架构规范               │
│  3. 修改单个组件       该组件上下文 + 组件内源码引用         │
│  4. 修改跨域逻辑       多个领域上下文 + 接口定义             │
│  5. 重构               领域边界 + 依赖关系 + 迁移策略        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 原则三：建立"上下文索引"而非"上下文本身"

**核心思想**：索引要轻量、可导航；详细内容按需加载

```
.asdm/contexts/
├── index.md                    # 超级精简（< 50 行）
│   ├── 项目概览（一段话）
│   ├── 技术栈列表（3-5 行）
│   ├── 模块索引表（表格）
│   └── 导航链接
│
├── 领域索引/                    # 每个领域的入口（轻量）
│   ├── 用户域.md               # < 30 行，核心概念 + 子模块链接
│   ├── 订单域.md
│   └── 支付域.md
│
└── 领域详情/                    # 按需加载的详细内容
    ├── 用户域/
    │   ├── index.md            # 详细索引
    │   ├── 实体模型.md
    │   ├── 服务接口.md
    │   └── 业务流程.md
    └── ...
```

## 5. 针对 200 万行代码库的具体建议

### 5.1 三明治架构

```
┌─────────────────────────────────────────────────────────────┐
│                    三明治上下文架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  顶层：全局索引（极轻量）                            │   │
│  │  - 项目一句话描述                                    │   │
│  │  - 技术栈（5 行以内）                                │   │
│  │  - 领域列表 + 入口链接                               │   │
│  │  大小：< 2KB                                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                           ↓                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  中层：领域索引（轻量，按需加载）                    │   │
│  │  - 领域一句话描述                                    │   │
│  │  - 子模块列表 + 入口链接                            │   │
│  │  - 关键依赖关系                                      │   │
│  │  大小：每个领域 < 1KB                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                           ↓                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  底层：详细内容（执行时按需加载）                    │   │
│  │  - 实体定义、API详情、流程图等                       │   │
│  │  - 由顶层/中层索引引导加载                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 粒度控制黄金数字

| 指标 | 建议值 | 理由 |
|------|--------|------|
| 单个上下文文件 | < 5KB | 保持信息密度，避免注意力分散 |
| 单次加载总量 | < 50KB | 留出空间给对话和源码 |
| 模块索引层级 | 2-3 层 | 太多层会迷失，太少会太粗 |
| 领域数量 | 5-10 个 | 符合人类认知负荷 |

### 5.3 构建策略对比

```
❌ 低效策略：全量构建
/初始化索引 → 解析全部200万行 → 生成全部上下文 → 50个文件 × 平均10KB = 500KB

✓ 高效策略：渐进式构建
/初始化索引（生成轻量索引） → 
  首次访问某领域时 → 生成该领域的轻量索引 → 
  执行具体任务时 → 按需生成详细内容
```

## 6. 具体实现模板

### 6.1 顶层索引模板（极简版）

```markdown

## 项目概览
一句话描述项目是什么。

## 技术栈
- 核心框架: xxx
- 主要语言: xxx
- 数据库: xxx

## 领域列表

| 领域 | 描述 | 入口 |
|------|------|------|
| 用户域 | 用户相关业务 | [.asdm/contexts/domains/user/](link) |
| 订单域 | 订单相关业务 | [.asdm/contexts/domains/order/](link) |
| ... | ... | ... |

## 架构规范
[standard-coding-style.md](link) | [architecture.md](link)

## 构建指南
[deployment.md](link)
```

### 6.2 领域索引模板（轻量版）

```markdown
# [领域名] 域

## 核心概念
一句话描述这个领域做什么。

## 子模块

| 模块 | 职责 | 关键类/文件 |
|------|------|-------------|
| 模块A | xxx | `com.xxx.A` |
| 模块B | xxx | `com.xxx.B` |

## 外部依赖
- 依赖领域X的[接口定义](link)

## 入口
[详细内容索引](./index.md)
```

### 6.3 AI 使用指南

```markdown
## 上下文使用原则

1. **启动时**：只读顶层 index.md（< 2KB）
2. **规划时**：根据任务涉及的领域，读对应领域索引（每个 < 1KB）
3. **执行时**：遇到具体实现，再读详细内容（< 5KB）
4. **跨域任务**：最多同时加载 2-3 个领域索引
```

## 7. 核心结论

```
合理上下文粒度 = f(任务类型, 领域边界, 信息密度)

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   任务粒度  ←→  上下文粒度                                  │
│       ↑                                                   │
│       └──────── 信息密度最大化 ────────┘                   │
│                                                             │
│   核心原则：                                                │
│   1. 索引要轻（< 5KB），内容要精（高密度）                   │
│   2. 按"决策域"划分，而非按"代码结构"划分                   │
│   3. 渐进式构建，按需加载                                    │
│   4. 最多 2-3 层索引，3 层以内找不到的内容直接读源码          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：

> 上下文的目的不是让 AI 知道一切，而是让 AI 在需要时能快速找到相关信息。200 万行代码库不需要全部"注入"，而是需要一个**高效的导航系统**。
