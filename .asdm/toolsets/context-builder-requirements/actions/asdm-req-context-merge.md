# ASDM 需求分析上下文融合 Action

> **用途**：将前端上下文和后端上下文融合为统一的业务上下文
>
> **触发词**：`/asdm-req-context-merge`

---

## 执行步骤

### Step 1: 确定输入来源

```
输入目录：
├── 前端上下文目录：.asdm/contexts/frontend-requirements/ 或项目中的前端上下文
└── 后端上下文目录：.asdm/contexts/backend-requirements/ 或项目中的后端上下文

输出目录：
└── .asdm/contexts/merged-requirements/     # 新目录，不修改原有文件
```

### Step 2: 读取并分析上下文

1. 读取前端 `index.md`，获取前端领域列表
2. 读取后端 `index.md`，获取后端领域列表
3. 读取各领域上下文文件

### Step 3: 建立领域映射

分析前后端领域的对应关系：

| 前端领域 | 后端领域 | 映射关系 | 统一命名 |
|----------|----------|----------|----------|
| user-module | user-domain | 1:1 | user-domain |
| order-module | order-domain | 1:1 | order-domain |
| home-module | - | 前端独有 | home-module |
| - | payment-domain | 后端独有 | payment-domain |

**映射规则**：
- 1:1 映射 → 直接合并
- 前端独有 → 单独保留
- 后端独有 → 单独保留
- 多:1 或 1:多 → 合并到目标域

### Step 4: 生成融合上下文

#### 4.1 创建输出目录结构

```
.asdm/contexts/merged-requirements/
├── index.md                    # 统一全局索引
└── domains/
    ├── [统一领域名]/
    │   ├── index.md            # 领域索引
    │   └── context.md          # 融合后的上下文
    └── ...
```

#### 4.2 生成统一全局索引

`index.md` 内容结构：

```markdown

> **后端来源**：[后端上下文路径]

**项目名称**：[项目名]

## 2. 统一业务域

|----------|------|------|
| [user-domain] | 前端+后端 | 用户相关功能 |

| [payment-domain] | 后端独有 | 支付服务 |

| [user-module] | [user-domain] | 完整映射 |

### 后端 → 前端映射

## 5. 导航指引

| 内容类型 | 文件路径 |
|----------|----------|
| [领域名] 详情 | [domains/xxx-domain/index.md] |
| ... | ... |

```

#### 4.3 生成领域上下文

## 完整使用流程

```bash
# 1. 发现领域（前端）
/asdm-req-domain-discovery   # 选择：前端

# 2. 发现领域（后端）
/asdm-req-domain-discovery   # 选择：后端

# 3. 构建前端上下文
/asdm-req-domain-context-build 用户模块
/asdm-req-domain-context-build 订单模块
# ...

# 4. 构建后端上下文
/asdm-req-domain-context-build user-domain
/asdm-req-domain-context-build order-domain
# ...

# 5. 融合上下文（新增）
/asdm-req-context-merge

# 6. 后续使用融合后的上下文
# 读取 .asdm/contexts/merged-requirements/ 进行需求分析
```

---

## 注意事项

1. **不修改原文件**：融合操作不会修改前端或后端的原有上下文
2. **统一命名**：融合时会建立统一的领域命名规范
3. **内容完整**：融合后的上下文包含前后端的全部关键信息
4. **大小控制**：单个领域上下文 < 10KB，超出时拆分为多个文件
