# ASDM Toolset - Context Builder Requirements

toolset-id: context-builder-requirements
toolset-name: Context Builder Requirements (需求分析上下文构建器)
version: 0.2.0
updated-date: 2026-04-24
toolset-description: 适配需求分析的上下文构建工具，支持前后端项目，从代码中抽取系统能力概览，为需求澄清和特性拆分提供系统上下文。

---

## 1. 概述

### 1.1 核心定位

**需求分析上下文**是专为需求分析场景设计的上下文内容，与代码导航上下文有本质区别：

| 上下文类型 | 目标 | 使用方式 |
|-----------|------|----------|
| 代码导航上下文 | 找到具体代码文件 | 根据 L2 导航读源码 |
| **需求分析上下文** | **理解系统能力** | **直接作为上下文给需求分析使用** |

### 1.2 前后端兼容

本工具集同时支持后端代码库和前端代码库：

| 项目类型 | 支持框架 | 代码特征 |
|----------|----------|----------|
| **后端项目** | Java (Spring Boot)、Go、Node.js | Controller、Service、Repository |
| **前端项目** | Vue、React、Taro | Views/Pages、Router、Store、API |

**前端/后端内容差异**：

| 维度 | 后端描述重点 | 前端描述重点 |
|------|-------------|-------------|
| 业务能力 | Service 方法、API 接口 | 页面功能、组件能力 |
| 业务流程 | 后端处理流程 | 页面交互流程 |
| 数据模型 | 数据库实体 | 页面数据结构、API 响应 |
| 业务规则 | 后端验证规则 | 前端验证规则、交互限制 |
| 用户角色 | 后端权限控制 | 前端路由权限、页面可见性 |

### 1.3 为什么需要需求分析上下文？

当团队使用需求分析 action 进行 AI 需求澄清、特性拆分时：

```
┌─────────────────────────────────────────────────────────────────┐
│  现状痛点                                                         │
│                                                                 │
│  AI 进行需求分析时：                                             │
│  - 不了解系统已有能力 → 容易设计重复功能                          │
│  - 不清楚业务流程 → 容易设计不合理流程                            │
│  - 不知道数据约束 → 容易设计无法实现的需求                        │
│  - 不了解用户角色 → 容易忽略权限控制                              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  解决方案                                                         │
│                                                                 │
│  预先构建需求分析上下文：                                         │
│  - 业务域能力 → 让 AI 知道系统能做什么                            │
│  - 业务流程 → 让 AI 知道业务如何运转                             │
│  - 数据模型 → 让 AI 知道有哪些数据                               │
│  - 业务规则 → 让 AI 知道有哪些限制                               │
│  - 用户权限 → 让 AI 知道有哪些用户类型                           │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 上下文分层架构

### 2.1 二维分层：维度 × 领域

```
┌─────────────────────────────────────────────────────────────────┐
│                    二维分层架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│           │  能力    │  流程   │  数据   │  规则   │  角色   │
│  ─────────┼─────────┼────────┼────────┼────────┼────────│
│  用户域    │   ✓    │   ✓   │   ✓   │   ✓   │   ✓   │
│  订单域    │   ✓    │   ✓   │   ✓   │   ✓   │   ✓   │
│  支付域    │   ✓    │   ✓   │   ✓   │   ✓   │   ✓   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**切分依据**：基于 **Domain（业务域）** 切分

### 2.2 层级设计

| 层级 | 文件 | 大小限制 | 说明 |
|------|------|----------|------|
| L0 | `index.md` | < 2KB | 全局索引，建立全局认知 |
| L1 | `domains/{name}/index.md` | < 3KB | 域内索引，导航作用 |
| L2 | `domains/{name}/context.md` | < 10KB | 域内全维度上下文 |

```
    │       # └── 用户角色
    │
    ├── order-domain/                     # 订单域
    │   └── ...
    │
    └── payment-domain/                   # 支付域
        └── ...
```

## 4. 工具集命令

| 命令 | 优先级 | 说明 |
|------|--------|------|
| `/asdm-req-domain-discovery` | 推荐首次使用 | 发现并定义项目业务域 |
| `/asdm-req-context-init` | 必须 | 初始化全局索引 |
| `/asdm-req-domain-context-build <领域名>` | 必须 | 为指定领域构建全维度上下文 |
| `/asdm-req-context-merge` | 前后端项目使用 | 融合前后端上下文为统一上下文 |
| `/asdm-req-context-update` | 按需 | 变更时更新上下文 |

## 5. 工作流程

### 5.1 单项目工作流

```markdown
## 步骤 1: 发现项目领域（推荐首次使用）
/asdm-req-domain-discovery
→ 分析代码结构，识别业务域
→ 生成领域发现报告

## 步骤 2: 初始化全局索引
/asdm-req-context-init
→ 生成 L0: index.md

## 步骤 3: 为每个领域构建上下文
/asdm-req-domain-context-build 用户域
/asdm-req-domain-context-build 订单域
/asdm-req-domain-context-build 支付域
→ 每个领域生成 L1 + L2 上下文

## 步骤 4: 后续维护
/asdm-req-context-update <领域名>
→ 按需更新指定领域的上下文
```

### 5.2 前后端分离项目工作流

```markdown
## 步骤 1: 发现前端领域
/asdm-req-domain-discovery   # 选择：前端
→ 分析前端代码结构，识别前端模块
→ 输出：前端领域列表

## 步骤 2: 发现后端领域
/asdm-req-domain-discovery   # 选择：后端
→ 分析后端代码结构，识别后端领域
→ 输出：后端领域列表

## 步骤 3: 初始化前端上下文目录
/asdm-req-context-init       # 输出到前端目录
→ .asdm/contexts/frontend-requirements/

## 步骤 4: 构建前端上下文
/asdm-req-domain-context-build 用户模块
/asdm-req-domain-context-build 订单模块
→ ...

## 步骤 5: 初始化后端上下文目录
/asdm-req-context-init       # 输出到后端目录
→ .asdm/contexts/backend-requirements/

## 步骤 6: 构建后端上下文
/asdm-req-domain-context-build user-domain
/asdm-req-domain-context-build order-domain
→ ...

## 步骤 7: 融合上下文
/asdm-req-context-merge
→ 读取前端和后端上下文
→ 建立领域映射关系
→ 生成统一上下文到新目录
→ .asdm/contexts/merged-requirements/

## 步骤 8: 使用融合后的上下文进行需求分析
```

### 5.3 执行次数对比

| 场景 | 步骤 | 执行次数 |
|------|------|----------|
| **单项目** | 初始化 | 1 次 |
| | 构建领域上下文 | N 次（N = 领域数量） |
| | **总计** | **1 + N 次** |
| **前后端分离** | 初始化（前端） | 1 次 |
| | 构建前端上下文 | M 次（M = 前端模块数） |
| | 初始化（后端） | 1 次 |
| | 构建后端上下文 | K 次（K = 后端领域数） |
| | 融合 | 1 次 |
| | **总计** | **2 + M + K + 1 次** |

## 6. 前后端上下文融合

### 6.1 为什么需要融合？

前后端分离项目中，前端和后端的领域划分往往不同：

```
┌─────────────────────────────────────────────────────────────────┐
│  前端模块划分              后端领域划分                          │
├─────────────────────────────────────────────────────────────────┤
│  user-module        ↔     user-domain                           │
│  order-module       ↔     order-domain                         │
│  home-module        →     [无对应后端域]                        │
│  cart-module        ↔     cart-domain                          │
│  [无对应前端模块]    ←     payment-domain                       │
│  [无对应前端模块]    ←     notification-domain                  │
└─────────────────────────────────────────────────────────────────┘
```

**融合目标**：建立统一的业务领域视图，包含前后端全部内容。

### 6.2 融合策略

| 映射类型 | 处理方式 | 示例 |
|----------|----------|------|
| 1:1 映射 | 直接合并 | user-module ↔ user-domain |
| 前端独有 | 单独保留 | home-module |
| 后端独有 | 单独保留 | payment-domain |
| 多:1 映射 | 合并到目标域 | m1+m2 → backend-domain |

### 6.3 融合后目录结构

```
.asdm/contexts/merged-requirements/
├── index.md                    # 统一全局索引
│   ├── 统一业务域列表
│   └── 领域映射关系
│
└── domains/
    ├── user-domain/           # 统一领域
    │   ├── index.md           # 领域索引
    │   └── context.md         # 融合后的上下文
    │       ├── 前端能力
    │       ├── 后端能力
    │       └── 前后端映射
    │
    ├── home-module/           # 前端独有领域
    │   └── ...
    │
    └── payment-domain/        # 后端独有领域
        └── ...
```

### 6.4 重要原则

> **不修改原文件**：融合操作不会修改前端或后端的原有上下文，所有融合结果输出到新目录。

## 7. 前后端项目识别

### 6.1 识别依据

```bash
# 前端项目特征
- 存在 src/views 或 src/pages 目录
- 存在 src/router 或 src/routes 目录
- 存在 src/components 目录
- 存在 package.json
- 主要文件后缀：.vue, .tsx, .jsx

# 后端项目特征
- 存在 src/main/java 或 src 下的 .java 文件
- 存在 pom.xml 或 go.mod
- 主要文件后缀：.java, .go
- 存在 Controller, Service, Repository 目录
```

### 6.2 代码分析策略

#### 后端项目

| 代码层 | 分析内容 | 上下文价值 |
|--------|----------|-----------|
| Controller | API 接口定义 | 能力入口 |
| Service | 业务逻辑 | 能力实现 |
| Entity/Model | 数据结构 | 数据模型 |
| Validator | 验证规则 | 业务规则 |

#### 前端项目

| 代码层 | 分析内容 | 上下文价值 |
|--------|----------|-----------|
| Pages/Views | 页面功能 | 能力入口 |
| Router | 路由配置 | 页面关系、权限 |
| Store/Pinia | 状态管理 | 业务状态 |
| API 层 | 接口调用 | 数据操作 |
| Components | 组件能力 | 可复用功能 |

## 7. 使用方式

### 7.1 需求分析 Action 配置

```yaml
# 需求分析 action 配置示例
actions:
  requirements_analysis:
    context_sources:
      - path: .asdm/contexts/requirements/index.md
      - path: .asdm/contexts/requirements/domains/{relevant_domain}/context.md
```

### 7.2 AI 使用上下文的方式

```
┌─────────────────────────────────────────────────────────────────┐
│  AI 进行需求分析时的工作流                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户发起需求分析请求                                            │
│      ↓                                                          │
│  读取 index.md → 了解项目概览和业务域列表                         │
│      ↓                                                          │
│  根据需求涉及的业务域，读取该域的 context.md                      │
│      ↓                                                          │
│  context.md 包含该域的所有维度信息：                              │
│  - 业务能力（页面功能 / API 能力）                                │
│  - 业务流程（交互流程 / 处理流程）                                │
│  - 数据模型（页面数据 / 数据实体）                                │
│  - 业务规则（前端规则 / 后端规则）                                │
│  - 用户角色（路由权限 / API 权限）                                │
│      ↓                                                          │
│  基于上下文进行需求澄清、特性拆分                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
